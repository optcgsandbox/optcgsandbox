## MINOR

- **RESIGN illegal during `trigger_window` for BOTH players** — `shared/engine/rules/legality.ts:17-23`. Early-return for `phase === 'trigger_window'` short-circuits to `[]` (non-controller) or only `RESOLVE_TRIGGER` actions (controller) BEFORE reaching the universal `actions.push({ type: 'RESIGN' })` at `legality.ts:42`. Effect: while damage is suspended on a flipped trigger, neither player can concede. Worker validates against `getLegalActions` (`worker/GameRoom.ts:100`), so a RESIGN message would be rejected as `illegal_action`. AFK on the trigger prompt = game stuck until DO grace timer fires (hook noted but unimplemented at `worker/GameRoom.ts:129`). Introduced by Phase A (whole `trigger_window` branch is new). Remediation: append `{ type: 'RESIGN' }` to the returned arrays in both `trigger_window` branches (lines 18 and 20-22), or restructure to fall through to the universal append at line 42.

## NO-FINDING (checked + cleared)

- **DON array shift/pop safety** — every `donCostArea.shift()!` / `donRested.shift()!` / `attachedDon.shift()!` is gated by a prior length check or `while (…length > 0)` loop. Verified: `applyAction.ts:75,79,89,117,126,256,291`; `phases/turn.ts:21,42,61,80-88`; `cards/effects/templates.ts:47,65,118`. No unguarded shifts on the engine boundary.
- **RESOLVE_TRIGGER non-controller no-op** — `applyAction.ts:321-322` rejects when `state.pendingTrigger.controller !== player`. Test coverage at `__tests__/trigger.test.ts:132-147`. Cannot be bypassed by the attacker activating the defender's trigger.
- **`pendingTrigger` SNAPSHOT leak** — concern was that `pendingTrigger.lifeCardInstanceId` reveals the flipped card identity to the attacker before the defender resolves. SNAPSHOT serialization (`worker/GameRoom.ts:154-164`) sends full state including the `instances` map; the lifeCardInstanceId resolves to a `cardId` the attacker can already read in the `instances[…]` keyset (broadcast since `9d75ba0`). The "full state to both seats" leak is **pre-existing** (`void seat` comment at `GameRoom.ts:157` predates Phase A); pendingTrigger inherits the same envelope, not a new exfiltration channel. Pre-existing per CLAUDE.md §3 criterion 5. Phase A did not regress visibility.
- **TriggerPrompt focus / Escape** — `src/components/TriggerPrompt.tsx:55-72` Tab/Shift+Tab cycles Activate ↔ Decline; auto-focus on Activate after open (`:42-49`). No Escape handler — by design: engine is suspended in `trigger_window` until `RESOLVE_TRIGGER` dispatches, so a UI-only Escape would orphan engine state. Decline button provides keyboard exit. `role="dialog"` + `aria-modal="true"` + `aria-labelledby` correct.
- **LifeRevealOverlay / EventCardOverlay unbounded re-render** — `src/components/LifeRevealOverlay.tsx:38-52` and `src/components/EventCardOverlay.tsx:42-58` both monotonically advance `lastProcessed` after every match, then break out of the loop (`return` at `LifeRevealOverlay.tsx:46`, `EventCardOverlay.tsx:51`). One event surfaced per effect run; history append-only. No infinite-render path even with crafted history. `setTimeout` dismissal cleared on unmount. Safe.
- **donArm XSS / user-controlled DOM** — `src/store/donArm.ts:23-27` holds an instance ID string only. Rendered via `data-don-instance={instanceId}` and `aria-label` (text node) in `CostAreaStrip.tsx:71-86`. No `dangerouslySetInnerHTML` anywhere in the repo (0 matches in src/, shared/, worker/). IDs come from the engine (`i0`, `i1`, …, minted at `GameState.ts:153`), not user input. No injection surface.
- **EventCardOverlay shows both players' events** — by intent (`EventCardOverlay.tsx:30` "Both players see it"). Matches rules-reference.md §1.5 (events resolve publicly). Not a leak.
- **`card.effectText` rendering** — rendered as React children (`TriggerPrompt.tsx:113`, `EventCardOverlay.tsx:99`), auto-escaped. Cards sourced from engine `cardLibrary`, never client input. No XSS.
- **No new secrets / tokens in client bundle** — `.env` unchanged, no new `import.meta.env.*` or hardcoded keys in any of the 9 files Phase B touched. `git show 937eb34 5ec8114 -- '*.env'` empty.
- **DON ATTACH UI dispatch path** — `donArm` exposes arm/disarm, but no component in `src/` actually dispatches `ATTACH_DON` on the second tap yet (`rg "ATTACH_DON" src/` returns only docstrings). UI is incomplete, not insecure — players cannot finish the attach gesture via UI. Server-side legality check at `legality.ts:76-89` still gates any future dispatch. Not a security finding.
