# Security audit — OPTCGSandbox v0 scaffold

- Commit: `ba15030` on `main`
- Date: 2026-05-28
- Scope: client-only scaffold (engine, protocol, Zustand store, UI shell). No `worker/`, no remote endpoints, no auth.
- Trust model at this stage: single-device hot-seat. Both "players" share the same JS heap, same `viewAs` toggle, same Zustand store. There is no adversary surface yet.

## MAJOR

## MINOR

## NO-FINDING — verified

### Identity / PII
- `optcgsandbox@proton.me` is NOT present in any shipped source or build artifact. Grepped `src/`, `shared/`, `index.html`, `vite.config.ts`, `package.json`, `dist/`.
- No author email, no owner handle, no git-user identifier embedded in the client bundle.

### Secrets in client bundle
- No `.env`, no `.env.local`, no `secrets/` directory in the repo. `.gitignore` covers `*.local` and `dist`.
- Grep for `sk-`, `tcg_`, `phc_`, `cfut_`, `sbp_`, `api_key`, `secret`, `password`, `token` across `src/`, `shared/`, `index.html`, `vite.config.ts`, `dist/` → zero hits.
- `vite.config.ts` defines no `define:` block injecting env vars into the client.

### Engine state filtering — hand leakage
- `GameState.players.{A,B}.hand` carries only `instanceId` strings (`GameState.ts:42`), but `cardLibrary` + `instances` are global and resolvable.
- For the current single-device hot-seat model this is intentional and correct: there is no "opponent client" to leak to. The UI's `PlayerSide` (PlayerSide.tsx:64-85) gates rendering via `showHand={false}` for the inactive side; `App.tsx:64-70` passes the inactive player with `showHand={false}`.
- A `serializeForPlayer` filter is referenced in GameState.ts:42 as a future requirement. This becomes BLOCKER-class the moment `worker/` exists and state is sent over the wire — not now. Per audit criterion #5 (introduced by current code) and the brief's explicit "multiplayer threats are out of scope until worker/ exists", deferred.

### Random / seed predictability
- Seed source: `Date.now() & 0xffffffff` (`game.ts:70`, `game.ts:96-97`).
- Mulberry32 PRNG (`Random.ts:14-19`) is non-cryptographic and deterministic from seed. Correct choice for replay/reconciliation per `backend-architecture.md §1`.
- For hot-seat, both players observe the same shuffle from the same seed on the same device — no fairness violation possible.
- `mulligan()` derives a child seed via `state.seed ^ 0xa1a1` / `0xb1b1` (setup.ts:34) — predictable, but again no adversary at v0.
- Server-authoritative shuffle (seed signed by Worker, revealed post-commit) is a `worker/` concern, not a v0 finding.

### Action validation — store does not call `getLegalActions` before dispatch
- `useGameStore.dispatch` (`game.ts:76-84`) forwards the action straight to `applyAction(state, state.activePlayer, action)` without consulting `getLegalActions`.
- `applyAction` itself enforces guards: `playCard` checks `inst.controller === player` (applyAction.ts:56), hand-membership (applyAction.ts:58-59), and `card.cost > p.donActive` (applyAction.ts:62). `attachDon` checks ownership (applyAction.ts:105). `resolveAttack` checks instance existence (applyAction.ts:122-124).
- Missing checks vs `getLegalActions` (rest state of attacker, "played this turn" / summoning sickness, character-vs-leader target legality, max-5 field with no replace target) are real correctness gaps but do not cross the security threshold at v0: there is no opponent, no rating, no economy, no leaderboard. A hot-seat user cheating against themselves is not a harm event.
- This becomes a BLOCKER the moment `worker/` validates client-submitted actions — the Worker MUST call `getLegalActions(state, player).some(eq(action))` before `applyAction`, not rely on `applyAction`'s partial guards. Flagged for the multiplayer milestone audit, NOT for v0.

### Action schema (`shared/protocol/actions.ts`)
- Zod `discriminatedUnion` correctly constrains shape. All `instanceId` fields typed `z.string()` with no length cap — unbounded string is reportable only when network-attacker exists. Not v0.
- No prototype-pollution risk in the Zod schema (no `z.record(z.any())`).

### Service Worker / PWA cache
- `vite.config.ts:30-39` caches `https://*.{png,jpg,jpeg,webp,svg}` via Workbox `CacheFirst`. No origin restriction.
- At v0 there are no remote image URLs being loaded — `quickDeck` (game.ts:28-41) generates synthetic card names, no `imageUrl` field rendered, no `<img>` in `CardChip.tsx`. The runtime caching rule is dormant.
- Becomes worth a closer look once card images from R2 / CDN ship; not exploitable today.

### Build / dependency surface
- `package.json` not deeply audited here (out of scope: pre-existing scaffold deps). `npm audit` belongs in CI, not this review.

## Summary
- Zero v0-scope security findings.
- Three items (state filtering for network play, seed signing, `getLegalActions`-before-dispatch on server) are correctly architected as deferred to `worker/` — they appear in `backend-architecture.md` / `security-architecture.md` and have placeholder hooks in code. None are exploitable in the current shipping artifact.
- Re-audit required when (a) `worker/` lands, (b) remote card images load, (c) any user-identifying field is added to the client.
